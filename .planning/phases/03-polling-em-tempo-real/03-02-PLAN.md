## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| DoctorDashboard → GET /api/turnos/meu/ | Endpoint autenticado (JWT) — dados de turno do médico logado |
| UPAManagerDashboard → GET /api/medicos/ + /api/turnos/ + /api/alertas/ | Endpoints autenticados (JWT) — dados da UPA do gestor |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-06 | Availability | Polling 15s em DoctorDashboard | accept | 15s é conservador; carga proporcional a médicos logados simultaneamente. Para TCC (baixo volume), sem risco. |
| T-03-07 | Availability | Polling 15s em UPAManagerDashboard (3 requests paralelos via Promise.all) | accept | 3 requisições a cada 15s por gestor logado. Volume de gestores é pequeno; impacto mínimo. |
| T-03-08 | Information Disclosure | Polling continua após logout se clearInterval não executar | mitigate | O `useEffect` cleanup (`return () => clearInterval(intervalo)`) é chamado automaticamente quando o componente desmonta. O logout destrói o componente autenticado, garantindo o cleanup. Verificado via testes (Test 2 de cada componente). |
| T-03-09 | Tampering | Resposta de polling com dados corrompidos altera estado do dashboard | accept | Os endpoints são autenticados via JWT com refresh rotation (apiFetch já implementa). Dados corrompidos retornariam 400/500 — o `catch {}` silencioso descarta o erro sem modificar o estado existente. |
| T-03-10 | Elevation of Privilege | Token JWT expirado durante sessão de polling | accept | apiFetch já implementa refresh automático com `pendingRefresh` singleton. Token expirado aciona refresh transparente. Se refresh falhar, `window.location.reload()` é chamado — comportamento seguro e já existente no projeto. |
</threat_model>
