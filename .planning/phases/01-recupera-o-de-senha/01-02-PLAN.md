};

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Django API | reset_token is user-supplied via URL param; passed directly to backend |
| Form input → API call | email and nova_senha are user-typed strings sent in POST body |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-02-01 | Tampering | RedefinirSenha — reset_token from URL | mitigate | Backend validates token expiry and single-use (Plan 01-01). Frontend passes it as-is; no client-side trust assumed. |
| T-01-02-02 | Information Disclosure | EsqueciSenha — email enumeration | accept | Backend returns same 200 response regardless of whether email exists (Plan 01-01 responsibility). Frontend never leaks existence. |
| T-01-02-03 | Denial of Service | EsqueciSenha — repeat submission spam | accept | Rate limiting is a backend concern; frontend disables button during isLoading to prevent double-submit. |
| T-01-02-04 | Spoofing | apiFetch — no hardcoded origin | mitigate | Uses `__API_URL__` injected at build time via Vite, not hardcoded localhost. Correct for Netlify production. |
</threat_model>
